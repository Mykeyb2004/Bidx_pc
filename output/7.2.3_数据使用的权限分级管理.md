# 7.2.3 数据使用的权限分级管理

#### 7.2.3 数据使用的权限分级管理
为严格贯彻落实《中华人民共和国网络安全法》、《中华人民共和国数据安全法》、《中华人民共和国个人信息保护法》以及民政部关于社会救助数据安全管理的系列规定，确保社会救助数据在合法、正当、必要的前提下得到安全、规范、高效使用，我方将构建一套以“岗位职责为基、数据敏感为纲、操作行为为目”的三维一体化权限分级管理体系。本体系旨在实现对数据访问的精细化、动态化、智能化管控，从源头上防范数据泄露、滥用、篡改等安全风险，确保每一项数据操作均权责清晰、有据可查、全程留痕。

一、权限分级管理的总体设计思路与核心原则

我方深知，社会救助数据涉及困难群众最核心、最敏感的个人隐私，其安全直接关系到群众的切身利益与人格尊严，关系到政府的公信力。因此，权限管理绝非简单的功能开关，而是一套融管理制度、技术手段、人员意识于一体的系统性工程。

（一）指导思想
以国家数据安全法律法规为最高准绳，以“最小必要授权”为核心理念，以保障社会救助对象合法权益为根本出发点，以业务流程顺畅高效为重要考量，构建一个既能满足各层级、各岗位工作人员履职需要，又能最大限度保护数据安全的权限管理闭环。

（二）核心原则
1. 合法合规原则：所有权限的设定、分配、变更和回收，均严格依据国家相关法律法规及本项目招标文件的明确要求。权限的范围和边界清晰界定，确保任何数据访问行为都在法律授权的框架内进行。
2. 最小必要原则：坚持“必需”和“最少”的授权标准。任何岗位和人员仅被授予其履行法定职责所必需的最小数据访问权限和操作权限。严禁“图方便”而授予超出岗位需求的宽泛权限，确保用户不能访问其工作职责之外的任何数据。
3. 权责对等原则：权限的授予与岗位职责、安全责任严格挂钩。谁申请、谁审批、谁使用、谁负责。通过权限体系明确每个数据使用者的责任范围，为后续的安全审计和责任追溯提供坚实基础。
4. 分级分类原则：对系统用户（按岗位、层级）和数据资源（按敏感度、业务属性）进行科学的分类分级，实施差异化的授权策略。高敏感度数据必须匹配高安全级别的访问控制，重要岗位必须经过更严格的授权审批。
5. 动态调整原则：权限并非一成不变。建立权限的定期审视与动态调整机制，当工作人员岗位变动、离职或职责范围发生变化时，系统管理员将依据规范流程，在24小时内及时调整或回收其权限，防止“影子权限”、“僵尸账户”的存在。
6. 全程留痕原则：对所有用户的权限申请、审批、变更以及所有对数据的访问、查询、修改、导出等操作行为进行全面、详尽的日志记录。审计日志不可删除、不可篡改，确保任何数据操作行为均可追溯到具体个人、具体时间、具体IP地址。

二、构建“岗位角色-数据敏感度-操作行为”三维一体的权限模型

为实现精细化管控，我方将摒弃传统粗放的权限管理模式，创新性地构建一个三维权限管控模型，将“人”（岗位角色）、“数据”（敏感度级别）和“行为”（操作类型）三个维度进行交叉矩阵授权。

（一）岗位角色维度（用户分级）
根据社会救助业务的实际流程和管理层级，将系统用户划分为以下典型角色，并可根据项目地的具体组织架构进行定制化调整：
1. 一线经办岗（乡镇/街道级）：主要负责救助申请的受理、信息录入、初步核查等基础性工作。此岗位是数据生产和初步使用的源头。
2. 审核复核岗（区/县级民政部门）：负责对一线提交的申请材料进行审核、复核，开展经济状况核对，组织民主评议等工作。
3. 审批决策岗（区/县级或市级民政部门）：负责对审核通过的申请进行最终审批，做出“批准”或“不批准”的决定，并对救助标准进行核定。
4. 资金发放岗（财务或指定业务科室）：负责根据审批结果生成资金发放清单，与财政、银行等部门进行对接，执行资金发放操作。
5. 动态管理岗（各级民政部门）：负责对在保对象进行定期复核、随机抽查，处理群众举报，更新救助对象信息。
6. 监督审计岗（纪检监察、内部审计部门）：为履行监督职责，需要对业务全流程或特定环节的数据进行查询、调阅，但原则上不具备修改权限。
7. 数据分析岗（政策研究室、信息中心）：为开展政策评估、趋势预测等工作，需要访问经过脱敏处理的宏观、中观数据，原则上不能接触未经处理的原始个人敏感信息。
8. 系统运维岗（技术支持方或信息中心）：负责系统的日常维护、故障排查、性能监控。严格遵守“运维不碰数据，数据不碰运维”的原则，其对服务器、数据库的后台访问权限与业务数据内容严格隔离。

（二）数据敏感度维度（数据分级）
依据《信息安全技术 个人信息安全规范》（GB/T 35273）等国家标准，结合社会救助业务特性，将系统内数据按敏感程度划分为以下级别：
1. 核心敏感数据（L3-最高级）：包括但不限于公民身份证号码、银行账号、家庭金融资产信息、残疾证号、健康状况与病历信息、联系方式等。对此类数据的访问将施加最严格的控制。
2. 重要敏感数据（L2-较高级）：包括但不限于家庭成员关系、就业情况、家庭具体住址、救助申请与审批过程记录、民主评议意见、信访举报内容等。
3. 一般业务数据（L1-普通级）：包括但不限于救助政策文件、办事指南、公示信息（按规定脱敏后）、系统操作日志、非关联到个人的统计数据等。
4. 公开数据（L0-公开级）：对外公开发布的政策法规、新闻动态、通知公告等。

（三）操作行为维度（权限类型）
定义用户可以对数据执行的原子操作，形成权限集：
1. 查询（Read）：查看数据内容。
2. 新增（Create）：录入新的数据记录。
3. 修改（Update）：对已有数据进行编辑和更新。
4. 删除（Delete）：逻辑删除或物理删除数据记录（高风险操作，需严格限制）。
5. 导出（Export）：将数据以文件形式（如Excel、PDF）从系统中导出。
6. 打印（Print）：调用打印机输出数据内容。
7. 审批（Approve）：对业务流程节点进行确认、通过或驳回的操作。
8. 授权（Authorize）：分配或管理其他用户权限的操作（仅限系统管理员）。

三、权限分级矩阵设计与典型应用场景解析

基于上述三维模型，我方将设计一张精细化的权限控制矩阵表，并固化到系统后台。以下为该矩阵的设计逻辑与应用场景示例：

（一）权限分级控制矩阵（逻辑示例）
1. 一线经办岗：
   - 对【核心敏感数据(L3)】：仅在“新增”救助申请时有录入权限；对自己经办的个案，在办理期间有受限的“查询”权限；严禁“修改”、“删除”、“导出”。
   - 对【重要敏感数据(L2)】：具备对自己经办个案的“新增”、“修改”（在特定流程节点前）和“查询”权限。
   - 操作限制：单次可查询的记录数受限，禁止批量导出；打印功能将自动添加包含操作者信息的水印。

2. 审核复核岗：
   - 对【核心敏感数据(L3)】：具备对其管辖范围内所有待审核个案的“查询”权限；不具备“新增”、“修改”、“删除”权限。
   - 对【重要敏感数据(L2)】：具备“查询”权限，可在审核环节填写复核意见（特定字段的“修改”权限）。
   - 操作限制：跨乡镇/街道的批量查询需额外授权；导出数据功能默认关闭，如确需使用，须发起线上审批流程，并记录在案。

3. 审批决策岗：
   - 对【核心敏感数据(L3)】：原则上仅能查看系统生成的、隐去部分敏感信息的审批摘要。如需查看原始信息，需触发二次身份验证（如短信验证码），并记录为高风险操作。
   - 对【重要敏感数据(L2)】：具备对其管辖范围内待审批个案的全面“查询”和“审批”权限。
   - 操作限制：主要操作为“审批”和“驳回”，不涉及数据内容的直接修改。

4. 监督审计岗：
   - 对【核心敏感数据(L3)】和【重要敏感数据(L2)】：具备广泛的、跨区域的“查询”权限，以满足审计需求。
   - 操作限制：严格限定为只读（Read-only），不具备任何写操作权限（新增、修改、删除）。所有查询行为均作为高级别审计事件被详细记录。

5. 系统运维岗：
   - 权限设定：绝对禁止其通过业务系统前端访问任何业务数据。其后台数据库操作权限仅限于数据库管理员（DBA）角色，且所有高危操作（如数据备份、恢复）需采用“双人复核”机制，即一人操作，另一人确认后方可执行。数据库访问日志将由独立的日志服务器采集，防止运维人员自删痕迹。

（二）典型应用场景解析
1. 场景一：低保申请入户核查
   - 乡镇经办人员小王使用其账号登录系统，系统自动将其角色识别为“一线经办岗”。他只能看到自己负责的网格/村居内的救助申请。在录入张三的申请信息时，他可以填写所有字段（新增权限）。提交后，在区级民政部门审核前，他可以修改非核心信息。一旦进入审核流程，他对自己录入的张三的信息将仅剩“查询”权限。

2. 场景二：数据批量导出用于统计分析
   - 市信息中心的数据分析员小李，为撰写年度救助分析报告，需要全市的救助人口结构数据。他登录系统，角色为“数据分析岗”。他无法直接访问任何具体救助对象的个人信息。他可以访问系统内置的“统计报表”模块，该模块的数据源已经过后台的匿名化处理（如将身份证号替换为不可逆的唯一ID，将姓名隐去，将家庭住址模糊到区/县一级）。他可以将统计结果“导出”为图表或Excel，但导出的文件中绝不包含任何可识别到个人的信息。

3. 场景三：工作人员岗位调动
   - 原在A街道担任“一线经办岗”的小张，被调动至区民政局任“审核复核岗”。人事部门在完成其调动流程后，会同步通知系统管理员。管理员通过后台权限管理模块，首先回收小张原有的“一线经办岗”角色权限，然后为其授予新的“审核复核岗”角色权限。整个过程在1个工作日内完成，并有详细的变更记录。小张再次登录系统时，其操作界面和数据访问范围已完全适配其新岗位职责。

四、技术实现与管理保障

为确保上述权限分级管理体系的刚性执行，我方将采取以下技术与管理措施：

（一）技术实现
1. 统一身份认证：建立全项目统一的身份认证中心，支持与政务统一认证平台对接，实现单点登录（SSO）。
2. RBAC与ABAC结合：以基于角色的访问控制（RBAC）为基础，实现权限与岗位的绑定。同时，引入基于属性的访问控制（ABAC）作为增强，实现更动态、更精细的访问判断（例如：可增加时间属性<仅工作时间访问>、IP地址属性<仅办公网络内访问>等）。
3. 最小权限原则的API设计：系统后台服务间的接口调用同样遵循最小权限原则，每个API接口仅暴露其完成业务所必需的数据字段。
4. 数据脱敏与水印：对于查询展示、打印、导出的敏感数据，系统将根据访问者的角色自动进行脱敏处理（如姓名显示“张*三”，身份证号显示“3201**********1234”）。所有打印和导出的文件将强制添加包含操作者信息、操作时间的明暗水印，以便追溯。

（二）管理保障
1. 建立制度规范：制定《社会救助信息系统权限管理办法》，明确各角色的职责边界、权限申请与审批流程、违规操作的处罚措施。
2. 定期权限审计：每季度由专人对所有用户的权限进行一次全面梳理和审计，清理不合理、不必要的权限，确保权限的“纯洁性”。
3. 强化安全培训：将数据安全和保密要求纳入所有系统用户的岗前培训和年度继续教育内容，通过案例教学、签订保密承诺书等方式，持续提升全体人员的安全意识。
4. 严格追责问责：对于违反权限管理规定，擅自越权操作、泄露或滥用救助数据的行为，一经发现，立即暂停其账号，并依据相关法规和单位制度追究其相应责任，构成犯罪的，依法移交司法机关处理。

综上所述，我方将通过周密的设计、先进的技术和严格的管理，构建一个安全、可靠、灵活、高效的数据使用权限分级管理体系，为本项目社会救助数据的全生命周期安全保驾护航，切实守护好每一位困难群众的“信息生命线”。
