# 代码安全风险报告

- 项目路径：`/Users/zhangqijin/PycharmProjects/BidX_simple`
- 审计时间：`2026-04-29 12:43:50 CST`
- 审计方式：只读代码审计、静态搜索、依赖/环境检查、临时目录证明性测试、现有测试套件验证
- 约束说明：本次审计未修改源码；报告不包含任何密钥原文

## 一、结论摘要

本项目是本地桌面/GUI 标书生成工具，传统 Web 攻击面较小，未发现 `eval`、`exec`、`yaml.load`、`pickle`、`shell=True` 等直接高危执行入口。当前主要安全风险集中在以下边界：

1. 本地密钥文件与 Git 跟踪边界。
2. 配置文件中的本地路径读取/写入边界。
3. LLM prompt、采购需求、评分标准、生成正文等敏感业务资料的 trace/debug/cache 落盘边界。
4. OpenAI 兼容外部模型服务的数据出境边界。

总体判断：**不存在明显远程代码执行风险，但存在较高的数据泄露与误配置导致的本机文件读写风险**。其中密钥文件进入 Git 跟踪、trace/debug 默认落完整上下文，是应优先处理的两类问题。

## 二、风险分级总览

| 编号 | 等级 | 风险点 | 影响面 | 优先级 |
|---|---|---|---|---|
| R-001 | 高 | 多份 `.env.local_*` 文件已被 Git 跟踪且包含非空 API Key | 密钥泄露、历史提交泄露 | P0 |
| R-002 | 高 | 配置路径可越过项目目录读取/写入本机文件 | 本地敏感文件进入 prompt、输出写到项目外 | P0 |

| R-004 | 中 | 外部模型服务地址完全由环境变量控制，缺少协议/host 信任约束 | 敏感资料发往非预期服务 | P1 |
| R-005 | 中 | 运行产物文件权限偏宽，默认按 `0644` 创建 | 同机用户可读敏感产物 | P1 |

## 三、详细发现

### R-001 高：密钥文件已进入 Git 跟踪

**现象**

审计发现以下密钥文件已被 Git 跟踪：

- `.env.local_qwen`
- `.env.local_oMlx`
- `.env.local.ssopen`
- `.env.local_gpt`
- `.env.local.codexzh.com`
- `.env.local.pack`

同时，检查结果显示这些文件中存在非空的 `BID_WRITER_*_API_KEY`。本报告只记录存在性，不记录密钥原文。

**证据**

- `.gitignore` 当前仅忽略 `.env.local` 和 `.env.*.local`，没有覆盖 `.env.local_qwen`、`.env.local.pack` 等命名：`.gitignore:13-15`
- `git ls-files` 显示上述 `.env.local_*` 文件已被跟踪。
- `ls -l` 显示这些文件权限为 `rw-r--r--`，即 `0644`。
- 检查命令只输出 `SET/EMPTY`，确认多项 API Key 为 `SET`。

**影响**

如果仓库被推送、共享、打包或备份，密钥可能已泄露。即使后续从工作区删除，历史提交仍可能保留原值。

**建议**

1. 立即轮换所有已进入 Git 跟踪的 API Key。
2. 使用 `git rm --cached` 将这些文件移出版本控制。
3. 如仓库曾对外共享，使用历史清理工具处理旧提交，并按泄露事件处理。
4. 将 ignore 规则收紧为更保守形式，例如忽略 `.env*`，再显式放行 `.env.example`。
5. 本地密钥文件权限改为 `0600`。

### R-002 高：配置路径可越过项目目录读写本机文件

**现象**

配置读取会把相对路径拼到配置目录或 `project.root_dir`，但没有在解析后强制限定路径必须位于项目根内。输出目录同样可配置到项目外，后续会自动创建目录并写入文件。

**关键代码位置**

- `bid_writer/config.py:226-247`：路径解析只拼接 base，没有做根目录约束。
- `bid_writer/config.py:249-258`：存在文件时直接读取文本。
- `bid_writer/config.py:302-330`：`bid_requirements`、`scoring_criteria` 等字段可从配置声明路径读取。
- `bid_writer/config.py:533-552`：`output_directory` 可解析为项目外目录。
- `bid_writer/file_saver.py:41-43`：输出目录会自动创建。
- `bid_writer/file_saver.py:357-366`：保存时直接写入目标路径。

**证明性测试**

使用临时目录构造配置：

- `bid_requirements_file: ../outside_secret.md`
- `output_dir: ../outside_output`

结果：

- `bid_requirements_read_marker=True`
- `output_directory_outside_project=True`
- `saved_outside_project=True`

这证明当前配置可以读取项目根外文件，并写出到项目根外目录。

**影响**

恶意或误配置项目可能把本机敏感文件读进 prompt，并发送给外部模型服务；也可能把生成结果写入用户未预期的位置。

**建议**

1. 对输入文件、输出目录、trace 目录、cache 目录分别做路径 allowlist。
2. 默认要求路径位于 `project.root_dir` 或配置文件目录下。
3. 如果确需外部路径，应增加显式开关和 GUI 二次确认。
4. 使用 `Path.resolve()` 后再校验 `is_relative_to(allowed_root)`。
5. 对符号链接也应按最终真实路径校验。

### R-003 高：trace/debug/cache 会落完整敏感上下文

**现象**

trace 默认写入 system prompt、user prompt、上下文拼装、生成正文和摘要。`redact_sensitive=true` 当前只影响 API base URL，不会脱敏 prompt、采购需求、评分标准、角色设定、用户附加要求和正文输出。

**关键代码位置**

- `bid_writer/generation_trace.py:78-90`：创建 trace 目录和产物路径。
- `bid_writer/generation_trace.py:125-142`：脱敏只处理 `api_base_url`。
- `bid_writer/generation_trace.py:145-154`：`additional_requirements` 会进入 heading payload。
- `bid_writer/generation_trace.py:227-237`：写入 manifest、heading、request options、context、system prompt、user prompt。
- `bid_writer/generation_trace.py:377-386`：写入生成正文和 summary。
- `bid_writer/context_pruner.py:854-892`：debug dump 会写入评分项、命中内容和章节上下文。
- `bid_writer/config.py:1379-1406`：trace 写 prompt/output/context 的默认值偏向开启。
- `config.example.yaml:69-79` 与当前业务配置均开启 trace/debug。

**证明性测试**

使用临时配置和 marker 验证：

- `SECRET_ROLE_MARKER` 出现在 `03_prompt_system.md`
- `SECRET_REQUIREMENT_MARKER` 出现在 `04_prompt_user.md`
- `SECRET_SCORING_MARKER` 出现在 `04_prompt_user.md`
- `SECRET_ADDITIONAL_MARKER` 出现在 `01_heading.json` 和 `04_prompt_user.md`

现有 `log/generation_traces/` 下也已有大量 `03_prompt_system.md`、`04_prompt_user.md`、`02_context_assembly.json`、`06_generation_output.md` 文件，权限多为 `0644`。

**影响**

投标资料、评分标准、采购需求、生成正文和内部写作规则会在本地长期留存。如果仓库被打包、备份、同步或同机其他用户可读，将造成业务资料泄露。

**建议**

1. 默认关闭 `runtime.trace.enabled`。
2. 即使开启 trace，也默认关闭 `write_prompt`、`write_context`、`write_output`。
3. `redact_sensitive=true` 时应对 prompt/context/output 中的敏感字段做规则化脱敏或摘要化保存。
4. trace/debug 目录权限使用 `0700`，文件权限使用 `0600`。
5. 增加“一键清理运行产物”和“打包前敏感文件扫描”。
6. 文档中明确 trace 属于敏感产物，不应提交或共享。

### R-004 中：外部模型服务缺少信任约束

**现象**

项目通过环境变量配置 OpenAI 兼容服务地址与密钥，主生成、辅助模型、embedding 都会向外部服务发送业务材料或其向量化文本。

**关键代码位置**

- `bid_writer/config.py:423-431`：主模型 `base_url` 和 `api_key` 来自环境变量。
- `bid_writer/ai_writer.py:92-99`：主生成客户端初始化。
- `bid_writer/embedding_store.py:22-31`：embedding 客户端初始化。
- `bid_writer/chapter_fact_extractor.py`、`fact_card_extractor.py`、`llm_verifier.py`、`h2_project_background.py` 也会在相应能力开启时调用模型服务。

**影响**

如果 `.env.local` 被误配置到不可信 host，采购需求、评分标准、正文、事实卡片等会被发送给非预期服务。当前代码没有对 `http`、公网 host、自签代理或 provider 做明确校验。

**建议**

1. 默认只允许 `https://`，`localhost/127.0.0.1` 可作为开发例外。
2. 在 GUI 中明确展示当前请求目标 host。
3. 增加 provider allowlist 或首次使用确认。
4. 对非 OpenAI 官方或组织内域名增加风险提示。

### R-005 中：敏感运行产物权限偏宽

**现象**

多个运行产物按系统默认 umask 创建。当前检查到大量 trace、output、embedding cache 文件为 `0644`。

**关键代码位置**

- `bid_writer/generation_trace.py:116-123`：trace JSON/Markdown 写入。
- `bid_writer/embedding_store.py:92-96`：embedding cache 写入。
- `bid_writer/chapter_fact_store.py:17-24`：章节 facts 写入。
- `bid_writer/timing_logger.py:31-34`：timing log 追加写入。

**影响**

同机其他用户或某些备份/同步进程可能读取敏感内容。对于投标资料、采购需求和生成正文，默认 `0644` 偏宽。

**建议**

1. 建立统一安全写文件 helper。
2. 敏感目录创建为 `0700`。
3. 敏感文件写入后设置 `0600`。
4. 对旧产物提供迁移/修复权限命令。

### R-006 低：Windows 输出目录打开方式可更稳妥

**现象**

Windows 分支使用字符串形式启动资源管理器：

- `bid_writer/gui.py:4248`

虽然没有 `shell=True`，直接命令注入风险有限，但路径中包含引号或特殊字符时行为可能不稳定。

**建议**

改为参数列表形式或使用 `os.startfile`。
