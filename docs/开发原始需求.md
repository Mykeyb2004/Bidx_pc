我想写一个智能扩写标书章节的智能体。场景是用于投标文件AI智能扩写H3/H4级章节内容。用户会输出一个h1-h4的md格式大纲+项目采购需求+评分标准。

我遍历H3/H4节点章节，让LLM根据完整的标书大纲+当期节点的信息+项目采购需求+评分标准+角色提示词来逐章节扩写。用户还会在扩写前加入一些自己的撰写要求。

目前看来我认为放入上下文中的内容太多。检索当前harness engineering的最近实践，给我提出建议。如何设计能更准确的完成这个任务。

---
### 大致想法
我查了截至 `2026-04-04` 的 OpenAI / Anthropic / Google 一手资料。把你这里的 “harness engineering” 理解为：不是继续堆 prompt，而是围绕模型调用去设计 `上下文选择 + 检索 + 约束 + 验证 + 反馈回路`。当前主流实践很一致：

不要把“完整大纲 + 全部采购需求 + 全部评分标准 + 角色提示词 + 用户要求”每次都塞进 H3/H4 调用。更稳的做法是：`小而稳定的固定前缀` + `针对当前章节的动态检索上下文` + `分阶段生成` + `自动评测重写`。OpenAI 2026-02-11 的 harness engineering 文章本质上也在强调这个方向：给 agent 一个“地图”，而不是一份巨大的说明书。

**我建议你这样设计**
- 先做 `预处理层`，不要直接写作。把输入拆成 4 类结构化对象：`大纲树`、`采购需求原子条目`、`评分点原子条目`、`用户附加要求`。
- 给每个 H3/H4 先生成一个 `section brief`，而不是直接出正文。这个 brief 最好是 JSON，字段至少包括：`section_id`、`章节路径`、`章节类型`、`本节目标`、`必须覆盖的评分点`、`必须引用的需求证据`、`禁写项`、`建议篇幅`、`风格约束`。
- 做 `章节类型路由`。不要一个 prompt 打天下。`技术方案`、`实施计划`、`服务保障`、`资质业绩`、`响应性条款` 这几类章节应该用不同 prompt/校验器。尤其 `资质/案例/证书/参数响应` 这类，必须强约束，尽量从结构化资料填充，少自由发挥。
- 每个 H3/H4 用 `两段式或三段式链路`：
  1. `planner`：抽取本节要点、证据、评分点覆盖计划  
  2. `writer`：按计划写正文  
  3. `judge`：检查覆盖率、事实依据、重复、空话、冲突，再决定是否重写
- 检索不要只做向量。你的场景里评分标准和采购需求经常有 `精确术语/编号/参数/关键短语`，所以更适合 `BM25/关键词 + 向量检索 + rerank` 的混合检索。Anthropic 的 Contextual Retrieval 对你尤其有用：先给 chunk 补上“它属于哪份文档、哪一章、对应什么评分点”的上下文，再做 embedding，检索会更准。
- 把 `评分标准` 变成“驱动写作的主索引”，不是普通参考材料。建议先把评分条款拆成原子评分点，再绑定到章节节点。这样模型不是“泛泛扩写”，而是在“为这个 H3/H4 最大化得分”。
- 做 `证据先行`。在正式扩写前，让模型先返回“本节使用哪些需求/评分证据”。Anthropic 的长上下文实践建议先抽相关引文再执行任务，这对防空话、防幻觉很有效。
- 中间产物全部用 `结构化输出`。让模型在中间阶段返回 JSON：`evidence_ids`、`score_coverage`、`risk_flags`、`missing_info`。自然语言只留给最终正文。
- 把 `已写章节记忆` 做成摘要账本，不要把已生成全文回灌给后续章节。每节写完，只保留 `关键承诺`、`核心术语`、`数字/日期`、`已覆盖评分点`、`与其他章节的依赖关系`。后续只按需取回全文。
- 如果同一个项目要跑很多 H3/H4，务必用 `prompt/context caching`。固定内容放前面，变量放后面。这样能省钱省时，但注意：`缓存解决成本，不解决噪声`，所以仍然要做检索裁剪。

**每次 H3/H4 调用，建议只放这些**
- `固定前缀`：短角色提示词、输出格式、文风规范、禁写规则
- `当前节点`：标题、层级、祖先路径、章节类型
- `邻域信息`：父节点全文 + 同级兄弟的一行摘要，不要塞兄弟全文
- `用户本次附加要求`：只保留规范化后的有效约束
- `检索证据`：本节 top-K 采购需求片段 + top-K 评分标准片段 + top-K 术语/案例/资质片段
- `全局一致性账本`：前文已承诺的关键事实、数字、术语

**你现在最该避免的 5 件事**
- 不要每节都喂完整大纲全文
- 不要把评分标准只当“背景材料”
- 不要让模型直接一跳生成最终正文
- 不要把企业资质、案例、参数响应交给模型自由编造
- 不要只凭“感觉写得像样”评估效果，必须上 eval

**最值得做的 eval**
- `评分点覆盖率`：每个原子评分点是否被明确响应
- `证据绑定率`：关键陈述是否能回溯到采购需求/评分标准/企业资料
- `重复率`：与相邻章节是否内容同质化
- `一致性`：数字、期限、服务承诺是否和前文冲突
- `空话率`：泛化套话占比
- `pairwise A/B`：全文塞入 vs 混合检索；一跳直写 vs planner-writer-judge

补一句工程判断：如果你的单项目材料其实不大，做一个 `全量上下文 + 缓存` 的 baseline 也值得保留。Anthropic 明确提过，小语料时“直接放全量上下文”有时是最简单方案。但对你这个标书场景，我更看好 `章节 brief + 混合检索 + 证据先行 + judge 重写`。
