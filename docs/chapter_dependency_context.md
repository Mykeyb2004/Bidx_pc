# 章节依赖关系与摘要注入

## 目标

为章节正文扩写增加一个“可见、可编辑、可复用”的依赖上下文层：

- 用户可为某个叶子章节手工选择多个依赖章节
- 系统保存这些依赖关系
- 生成时优先把依赖章节提炼为约 200 字摘要
- 摘要默认注入“附加扩写要求”，单章可编辑，批量按章节后台追加

这套机制服务于“避免跨章节重复、冲突和术语漂移”，但不改变 `AIWriter` 的主 prompt contract。

## 数据文件

默认保存在项目根目录下：

- `.bid_writer/chapter_dependencies.json`
- `.bid_writer/chapter_summaries.json`

### `chapter_dependencies.json`

使用 `目标章节 full_path -> 依赖章节 full_path 列表` 的结构保存。

设计原因：

- `title` 可能重名
- `full_path` 更适合作为项目内稳定键

### `chapter_summaries.json`

使用 `章节 full_path -> 摘要记录` 的结构保存。

摘要记录至少包含：

- `title`
- `source_kind`
  - `output`
  - `planned`
- `source_hash`
- `summary`
- `generated_at`

## 摘要来源

### `output`

当依赖章节已有正文文件时：

1. 读取最近一次保存的章节正文
2. 基于正文生成约 160-220 字摘要
3. 用正文内容 hash 做缓存命中判断

如果正文未变化，则直接复用旧摘要。

### `planned`

当依赖章节还没有正文时：

1. GUI 先提示用户是否现在提炼规划摘要
2. 若用户确认，则基于章节路径、同级边界、需求要点、评分关注生成一段规划摘要
3. 该摘要也会缓存，供后续复用

一旦该章节将来有了正文，运行时会优先升级为 `output` 摘要。

## GUI 行为

### 设置依赖

入口位于 GUI 的“设置章节依赖...”。

约束：

- 仅支持选中 1 个叶子章节后编辑
- 依赖章节支持多选
- 当前第一版只支持“叶子章节依赖叶子章节”
- 保存后主界面的章节树会立即重绘：
- 被依赖章节继续只显示 `🔗` 小图标
- 配置了依赖关系的章节会显示 `⇢N章`，表示当前章节依赖了多少其他章节
- 将鼠标悬停在 `⇢N章` 所在行时，会弹出提示，列出具体依赖章节

### 单章节生成

若目标章节配置了依赖关系，则系统会：

1. 解析可用的依赖章节摘要
2. 将摘要块默认填入“附加扩写要求”
3. 允许用户手工编辑这段内容
4. 若需要刷新正文摘要或补生成规划摘要，界面会弹出“正在后台检查/生成关联摘要”的忙碌提示，避免用户误以为程序无响应

### 多章节批量生成

批量模式下的推荐交互是：

- 对话框中的“附加扩写要求”只编辑共享要求
- 关联章节摘要不显示为共享文本
- 每个章节在实际发起生成前，再单独拼接自己的依赖摘要块

这样可以避免：

- A 章节的依赖摘要污染 B 章节
- 批量运行中前文新生成的正文无法被后文章节及时复用

## Prompt 注入方式

关联章节摘要当前仍归属于 `additional_requirements`，不新增独立 block。

典型注入格式：

```text
请参考以下关联章节摘要，保持术语、职责、承诺和章节边界一致，避免与关联章节重复或冲突。

- 人员配置方案：……
- 进度计划安排：……

请参考以上章节总结内容进行扩写。
```

## 维护边界

当前实现刻意保持以下边界：

- 不递归展开“依赖的依赖”
- 不自动猜测依赖关系，完全以用户选择为准
- 不修改 `AIWriter.build_prompt_result()` 的 block 结构
- 不把关联章节摘要并入 `context_pruner`

如果未来要继续演进，优先考虑：

1. 在工作区中显示当前章节的依赖摘要预览
2. 为依赖关系增加导入/导出或批量编辑能力
3. 在 trace 侧单独记录“章节依赖摘要块”的来源章节清单
