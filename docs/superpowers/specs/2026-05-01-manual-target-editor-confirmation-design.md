# 招标资料人工选择与目标编辑框确认设计

## 背景

当前“开始抽取”会把招标文件转换为 Markdown，并自动定位“项目采购需求”和“评分标准”。现有人工确认窗口已经要求用户确认后再写入目标文件，但窗口仍保留“块预览”思路，自动命中内容容易被用户误认为最终可直接保存的结果。

实际使用中，“项目采购需求”和“评分标准”的自动定位会受目录、章节标题、跨页表格和招标文件排版影响而偶发不准。新的设计把自动定位降级为导航提示，最终写入内容必须由用户在源文中选取并放入目标编辑框确认。

## 目标

- 取消确认窗口中的“块预览”作为主要交互入口。
- 保留自动定位能力，但只用于滚动到疑似位置和淡色高亮提示范围。
- 用户必须显式拖选源文，并点击“使用选区”把内容放入目标编辑框。
- 目标编辑框是保存的唯一来源；用户可在保存前编辑 Markdown。
- “项目采购需求”和“评分标准”继续按两步分步确认。
- 每一步点击“存入”后立即写入对应目标文件。
- 中途取消时，已存入文件保留，未存入文件不写入。

## 非目标

- 不重写自动章节定位算法。
- 不引入 PDF、Word 原版页面预览或页面框选。
- 不做多段追加、拼接或结构化重排；“使用选区”只替换目标编辑框全文。
- 不改动生成链路读取 `项目要求/项目采购需求.md` 和 `项目要求/评分标准.md` 的方式。

## 用户流程

用户在新建配置向导的“项目材料”步骤点击“开始抽取”。

系统执行现有转换与自动定位：

1. 将招标文件转换为 Markdown 和 block 列表。
2. 使用现有抽取器定位“项目采购需求”和“评分标准”的疑似范围。
3. 写入导入报告。
4. 打开人工确认窗口。

第 1 步为“项目采购需求”：

1. 源文区自动滚动到算法疑似位置。
2. 算法命中范围以淡色背景提示。
3. 目标编辑框初始为空，不自动填入算法命中内容。
4. 用户在源文区拖选需要保存的 Markdown。
5. 用户点击“使用选区”，选区内容替换目标编辑框全文。
6. 用户可在目标编辑框中删改、补标题或调整换行。
7. 用户点击“存入项目需求”，系统立即写入 `项目要求/项目采购需求.md`。
8. 窗口进入第 2 步。

第 2 步为“评分标准”，流程相同，最终写入 `项目要求/评分标准.md`。

如果用户取消窗口：

- 已点击“存入”的文件保留。
- 未点击“存入”的文件不写入。
- 向导提示确认未完成，并停留在项目材料步骤。

## 窗口结构

确认窗口保留三栏布局。

左侧控制栏：

- 当前步骤：`第 1/2 步 项目采购需求` 或 `第 2/2 步 评分标准`。
- 自动定位状态：显示“已跳到疑似章节”或“未自动定位，请手动选择”。
- 导航按钮：`上一章节`、`下一章节`，继续复用现有章节边界导航。
- 选区按钮：`使用选区`，把源文当前选区替换到目标编辑框。
- 保存按钮：`存入项目需求` 或 `存入评分标准`。
- 取消按钮。

中间源文区：

- 展示转换后的 Markdown 全文。
- 内容只读，但允许用户拖选文本。
- 自动命中范围使用淡色高亮。
- 用户当前拖选范围使用 Tk 文本选区显示。
- 不再展示或依赖块预览。

右侧目标编辑框：

- 可编辑 Markdown 文本。
- 每一步进入时默认为空。
- “使用选区”会替换目标编辑框全文。
- “存入”只读取目标编辑框内容。
- 用户可以直接在目标编辑框内编辑最终要写入的内容。

## 行为规则

自动定位：

- 若自动定位成功，源文区滚动到命中范围开始位置。
- 命中范围只作为提示，不进入目标编辑框。
- 若自动定位失败，源文区从全文开头显示，并提示用户手动选择。

使用选区：

- 源文区没有选区时，点击“使用选区”提示“请先在源文区选择文本”。
- 源文区有选区时，选区文本替换目标编辑框全文。
- 替换后目标编辑框获得焦点，方便继续编辑。

存入：

- 目标编辑框去除空白后为空时，禁止存入。
- 目标编辑框内容较短时，弹出确认提示，用户可继续或返回编辑。
- 项目采购需求缺少常见需求关键词时，弹出“可能不是项目采购需求”的确认提示。
- 评分标准缺少评分关键词且不是 Markdown 表格时，弹出“可能不是评分标准”的确认提示。
- 用户确认存入后立即写入对应文件，并沿用现有覆盖确认和 `.bak` 备份策略。

章节导航：

- `上一章节` 和 `下一章节` 只移动源文区选区或视图位置。
- 导航不会修改目标编辑框。
- 用户仍需点击“使用选区”才会把新选区放入目标框。

## 数据流

`TenderImportService` 继续保持服务层职责：

1. 转换招标文件。
2. 调用自动抽取器。
3. 写入抽取报告。
4. 调用人工确认回调。
5. 接收人工确认结果中的最终 Markdown。
6. 写入目标文件并处理覆盖备份。

`ManualTenderSectionSelection.markdown` 表示目标编辑框最终内容，而不是算法选区内容。`start_block_id` 和 `end_block_id` 只在最终内容能够精确对应一段源文 block 范围时保留；只要用户在目标编辑框里改动过内容，就置空并标记 `manually_adjusted=True`。

报告中继续区分：

- 自动抽取范围：算法找到的疑似章节。
- 人工确认内容：用户最终存入目标文件的 Markdown。
- 是否人工调整：源文选区或目标编辑框发生人工操作时为 true。

## 组件边界

### `bid_writer/tender_import_dialog.py`

主要修改点：

- 移除确认窗口中的“块预览”栏和块点击选择入口。
- 将中间栏改为只读 Markdown 源文选择区。
- 新增右侧目标编辑框。
- 新增“使用选区”按钮。
- 调整保存逻辑，使“存入”读取目标编辑框内容。
- 保留现有分步确认、章节导航、校验提示和取消流程。

### `bid_writer/tender_selection_model.py`

保留纯数据能力：

- block 到 Markdown 字符范围映射。
- 自动定位范围到源文提示范围的映射。
- 选区内容校验。

模型语义：

- 默认选择不再作为目标框初始内容。
- 自动定位范围和人工确认内容在模型语义上分离。

### `bid_writer/tender_import_service.py`

核心职责保持不变：

- 继续只接收人工确认结果里的 Markdown 并写文件。
- 不依赖 Tk 细节。
- 继续处理覆盖确认、备份、导入报告和取消结果。

### `bid_writer/new_config_wizard.py`

继续调用人工确认窗口：

- 导入完成后填充 `./项目要求/项目采购需求.md` 和 `./项目要求/评分标准.md`。
- 用户取消时保持在项目材料步骤，并提示确认未完成。

## 测试计划

使用 `uv run pytest`。

对 `tender_import_dialog` 的测试：

- 打开确认窗口时，目标编辑框为空。
- 自动定位成功时，源文区滚动或选中提示范围，但目标编辑框仍为空。
- 点击“使用选区”会用源文当前选区替换目标编辑框全文。
- 多次点击“使用选区”时，后一次选区替换前一次内容。
- 手动编辑目标框后，点击“存入”保存编辑后的内容。
- 目标编辑框为空时禁止存入。
- 章节导航不会修改目标编辑框。

对 `tender_selection_model` 的测试：

- 自动定位结果能映射为源文提示范围。
- 缺失自动定位时返回空提示范围。
- 校验函数能识别空内容、过短内容、疑似非需求内容和疑似非评分内容。

对 `tender_import_service` 的测试：

- 人工确认返回目标编辑框内容后，服务层写入该内容。
- 手动调整结果在导入报告中记录为 `manually_adjusted=True`。
- 用户取消时不写入未确认文件。
- 已确认写入的文件在后续取消时保留。
- 覆盖已有文件时继续生成 `.bak`。

## 验收标准

- 确认窗口不再出现“块预览”作为用户选择入口。
- 自动定位不会自动填充目标编辑框。
- 用户必须通过“使用选区”或直接编辑目标框确定待保存内容。
- “存入项目需求”和“存入评分标准”写入的是目标编辑框内容。
- 默认定位错误时，用户可在源文区选择其他范围并保存。
- 自动定位失败时，用户仍可从全文手动选择并保存。
- 逐步写入、覆盖备份、取消语义与现有人工确认流程保持一致。
