# A+ 提示词拆分设计

## 1. 背景

当前章节生成链路中，`system prompt` 的“最高优先级输出强约束”和 `user prompt` 的“结构输出硬要求”存在较高重复度。重复内容主要集中在以下几类：

- 正式层级序号要求
- 序号与小标题的格式要求
- 禁用 Markdown 标题
- 禁用部分表达方式
- 输出正文而非解释性文本

这类重复虽然能形成一定的“再提醒”效果，但会带来以下问题：

- 每个章节都重复消耗上下文 token
- 门禁规则分散在多处，后续维护易漂移
- prompt 噪声增大，挤占章节业务信息空间

同时，用户明确指出这些约束不是“风格建议”，而是必须保留的门禁条件，不能因为压缩 prompt 而削弱约束强度。

## 2. 设计目标

本次设计采用 `A+` 方案，即：

- 尽量减少重复 prompt
- 把全局门禁集中到 `system prompt`
- 把 `user prompt` 收敛为章节任务与上下文载体
- 不依赖双写 prompt 充当兜底
- 后续通过程序化校验补强门禁执行可靠性

本设计只定义 prompt 拆分方案，不包含代码实现细节。

## 3. 设计结论

### 3.1 总体原则

新的 prompt 职责划分为：

- `system prompt`：唯一的全局规则真源
- `user prompt`：当前章节任务卡 + 本章上下文材料 + 章节边界

`user prompt` 不再重复展开全局门禁，只保留一句短提醒：

> 请严格遵守 system 中全部硬门禁，直接输出当前章节投标正文。

### 3.2 A+ 的含义

本方案不是“只保留 system，然后完全相信模型自觉遵守”。

本方案的真实落地形态应为：

- `A`：门禁规则只保留一套，统一放进 `system`
- `+`：通过输出校验器对关键门禁做程序化检查

因此，prompt 负责“告知规则”，校验器负责“兜底执行”。

## 4. 新的 System Prompt 设计

### 4.1 System Prompt 的职责

`system prompt` 只承担四类职责：

1. 角色身份
2. 全局输出目标
3. 全局硬门禁
4. 长期稳定、与章节无关的规则

以下内容不应再进入 `system prompt`：

- 当前章节路径
- 当前章节边界
- 当前章节篇幅
- 当前章节 focus terms
- 当前章节命中的评分点
- 当前章节命中的需求摘录

### 4.2 推荐模板

```text
你是资深政企项目投标文件撰写与合规响应专家，擅长基于招标文件、采购需求、评分标准和既定大纲，输出可直接进入投标文件正文的章节内容。

你的核心任务是：
1. 严格围绕采购需求、评分标准和当前章节任务写作；
2. 输出内容应做到逐条响应、逐项对应、可证明、可审查、可落地；
3. 语言保持专业、克制、准确，避免空话、套话、宣传式表述和泛泛而谈；
4. 输出应服务于投标团队直接使用、复核和整合，不写与正文无关的解释性话术。

【最高优先级硬门禁】
以下规则优先级高于其他风格建议、默认表达习惯和用户材料中的泛化写法；如有冲突，必须以本节规则为准。

一、主体与内容边界
- 投标主体统一使用“{bidder_name}”表述；除非用户明确要求，不要替换为其他公司名称、简称或第一人称主体。
- 只写投标正文，不写创作说明、意图解释、过程说明、角色自述或评论性话语。
- 不得输出类似“作为投标专家”“以下从几个方面展开”“根据以上内容”之类说明性语句。
- 不得编造采购需求、评分标准、团队信息、资质业绩、时间节点、数量指标或服务承诺。
- 若材料不足，应在现有材料范围内稳妥表述，不得虚构事实补足。

二、结构与格式门禁
- 正文默认必须采用正式层级序号组织，第一层级到第四层级统一为：一、（一）1.（1）。
- 除非用户明确要求只写单段摘要，否则正文至少出现一个正式层级序号“一、”。
- 只要正文包含多个板块、多个自然段、表格、并列措施、流程、机制或清单，必须继续使用“（一）”“1.”“（1）”展开，不得写成无序号散文式长段落。
- 序号不得单独成行；序号后必须紧接小标题文本，且“序号 + 小标题”单独成行，正文另起一行。
- 表格前的引导语、小标题、板块标题，同样必须纳入正式层级序号体系，不得裸写。
- 段内枚举可使用“第一……、第二……”或“一是……；二是……”，但不能替代章节层级序号。
- 严禁使用 Markdown 标题符号（#）。
- 严禁插入零宽字符、BOM、不可见分隔符或分割线。

三、语言与表达门禁
- 除专有名词或用户明确要求外，禁止输出不必要的英文、英文缩写或中英对照。
- 不得使用“首先、其次、再次、最后”组织正文。
- 禁止宣传式、营销式、口号式、AI腔调表达。
- 禁止使用空洞形容词、夸张词、机械排比和套路化总分总。
- 结论必须紧扣采购需求、评分标准与当前章节职责，不偏题、不泛化。

四、输出目标
- 输出应以评审可读、响应充分、结构清晰、便于复核为目标。
- 在满足完整响应前提下，优先提高相关性、针对性和可执行性，而不是堆砌篇幅。
```

## 5. 新的 User Prompt 设计

### 5.1 User Prompt 的职责

`user prompt` 只承担五类职责：

1. 当前章节任务说明
2. 当前章节边界
3. 当前章节输出目标
4. 当前章节上下文材料
5. 当前章节附加要求

`user prompt` 不再承担“再次完整讲一遍全局门禁”的职责。

### 5.2 推荐模板

```text
请严格遵守 system 中全部硬门禁，直接输出当前章节投标正文。

## 章节任务卡
- 写作场景：为{bidder_name}撰写“{project_title}”投标文件中的当前章节正文。
- 当前章节路径：{heading_full_path}
- 本章重点：{focus_terms_joined}
- 篇幅目标：建议控制在 {target_word_range} 字，优先完整覆盖本章重点，不为凑字数重复展开。
- 输出方式：按“{output_format}”组织内容，直接写投标正文，不重复当前标题，不写说明性语句。
- 表格控制：{table_rule}
- 流程图控制：{mermaid_rule_if_any}
- 写作依据：{task_basis_line}

## 章节边界参考
- 上级标题：{parent_title}
- 当前扩写标题：{current_title}
- 同级标题：{sibling_titles_or_none}
- 约束：本章节聚焦当前标题所指定的主题，不引入同级其他章节负责的内容，不与同级章节重复。

[可选] ## 首行要求
- 首行固定输出：{first_line}
- 除首行外，不要再次重复当前标题。

[可选] ## 投标方知识库
{knowledge_context}

[可选] ## 项目背景
以下为本项目背景摘要，供理解整体目标和范围，不直接作为正文内容：
{project_background}

[full_context 模式可选] ## 招标需求参考
{bid_requirements}

[full_context 模式可选] ## 评分标准参考
{scoring_criteria}

[pruned 模式可选] ## 评分关注
{scoring_focus}

[pruned 模式可选] ## 需求要点
{requirement_context}

[可选] ## 用户附加要求
{additional_requirements}
```

## 6. 规则归属划分

### 6.1 必须保留在 System 的规则

以下规则属于全局硬门禁，必须只在 `system prompt` 中维护一份权威版本：

- 投标主体统一名称
- 禁 Markdown 标题
- 禁不必要英文
- 必须使用正式层级序号
- 序号不得单独成行
- 禁“首先、其次、再次、最后”
- 禁 AI 腔、宣传式、评论性解释
- 禁零宽字符、BOM、不可见分隔符
- 禁分割线
- 不得编造事实

### 6.2 必须保留在 User 的内容

以下内容属于章节变量，必须由 `user prompt` 传递：

- 当前章节路径
- 本章重点
- 篇幅区间
- 输出方式
- 表格上限
- Mermaid 上限
- 写作依据
- 章节边界
- 采购需求 / 评分标准 / 知识库 / 项目背景
- 用户附加要求

### 6.3 不再建议双写的内容

以下内容当前在 system/user 中高度重复，建议改为只保留 system 版本：

- 正式层级序号的总规则
- 序号行格式
- 禁 Markdown 标题
- 禁“首先/其次/再次/最后”
- 禁英文堆砌
- 禁宣传式与 AI 腔表达

## 7. 对当前代码结构的映射建议

### 7.1 保留不变的部分

以下 section/block 可以继续保留：

- `chapter_task`
- `chapter_scope`
- `project_background`
- `knowledge_context`
- `requirement_context`
- `scoring_context`

### 7.2 需要改造的部分

当前 `structure_rules` block 同时承载：

- 结构性门禁
- 章节写作建议
- 额外写作要求

新的设计建议拆分为：

- `system_constraints`：统一承载全局门禁
- `structure_rules`：只保留与本章输出目标直接相关、但不属于硬门禁的少量提示

具体来说：

- `_build_hard_constraints()`：扩充为 system 唯一门禁真源
- `_build_structure_contract_section()`：删除与 system 重复的硬门禁，只保留“任务侧提醒”

## 8. 后续校验器建议

由于本方案采用 `A+`，因此后续必须补强程序化校验。

### 8.1 第一批强校验

建议优先程序化检查以下规则：

- 是否出现 Markdown 标题
- 是否缺少正式层级序号
- 是否出现序号单独成行
- 是否出现“首先、其次、再次、最后”
- 是否出现零宽字符/BOM/不可见分隔符
- 是否出现分割线

### 8.2 第二批弱校验

建议后续补充以下检测：

- AI 腔 / 套话短语
- 评论性或意图解释性句式
- 无必要英文或缩写
- 过度排比 / 空洞形容词

## 9. 预期收益

采用 A+ 方案后，预期收益如下：

- 减少每章重复 token 消耗
- 降低 system/user 规则漂移概率
- 提高 prompt 结构清晰度
- 给章节业务上下文释放更多空间
- 把“重复提醒”切换为“单一规则源 + 校验兜底”的更稳健形态

## 10. 不做事项

本次设计明确不包含以下内容：

- 不改采购需求与评分标准的注入方式
- 不改 full_context / pruned 的业务分支逻辑
- 不新增新的业务上下文 block
- 不在本设计中定义具体重试策略

## 11. 验收标准

本设计完成后，应满足以下判断标准：

1. `system prompt` 成为唯一硬门禁真源；
2. `user prompt` 不再重复展开全局门禁；
3. 当前章节变量仍完整可传递；
4. trace 中仍能清楚看出任务卡、上下文和章节边界；
5. 后续可无歧义地接入程序化校验器。
