# 事实卡片库列表化与单卡编辑设计

## 背景

当前“事实卡片库”窗口同时展示卡片列表和下方整库编辑器。用户希望该窗口只保留卡片列表，减少窗口高度和认知负担；当用户双击列表记录或点击“编辑卡片”按钮时，再打开独立编辑弹窗。

## 目标

- 事实卡片库窗口只承担浏览和入口职责。
- 支持在库窗口中新建事实卡片。
- 支持通过双击列表记录或点击“编辑卡片”编辑已有事实卡片。
- 编辑已有卡片时允许修改名称、分类、作用域、约束、内容。
- 卡片来源信息保持只读并在保存时保留，不允许用户修改来源类型、章节路径或提炼要求。

## 交互设计

事实卡片库窗口显示当前全部卡片列表，保留名称、来源、作用域、约束、分类、内容列。窗口按钮为“新建卡片”“编辑卡片”“关闭”。未选中卡片点击“编辑卡片”时提示用户先选择记录。双击一行与点击“编辑卡片”行为一致。

新建和编辑都复用单卡编辑弹窗。新建时弹窗为空白草稿，默认 `scope=local`、`enforcement=reference`。编辑时弹窗加载当前卡片，并显示来源提示；保存后只更新该卡片可编辑字段，保留原始 `id`、`active`、`source`、`created_at`。

## 保存策略

现有 `save_library_cards()` 是整库覆盖式保存。为避免单卡编辑时误删其他卡片，主窗口统一通过“当前全部卡片草稿 + 当前单卡变更”的方式调用该接口：

- 新建卡片：将新草稿追加到现有全部卡片草稿后保存。
- 编辑卡片：用同 `card_id` 的新草稿替换现有草稿后保存。

后续如果事实卡片操作继续增多，可再下沉为 store 层 `upsert` 接口；本次不新增存储 API，降低改动面。

## 测试策略

- 单卡编辑弹窗应能接收初始草稿并返回保留 `card_id` 的草稿。
- 库窗口应只暴露列表操作结果，不再返回整库编辑结果。
- 主窗口编辑已有卡片时应保存完整卡片库，并保留被编辑卡片来源。
- 主窗口新建卡片时应追加到完整卡片库，不删除已有手工或提炼卡片。
