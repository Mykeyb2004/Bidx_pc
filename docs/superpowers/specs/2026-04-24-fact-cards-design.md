# 事实卡片设计

## 1. 背景

当前项目已经具备两类与“事实复用”相关的能力：

- `bid_writer/chapter_fact_extractor.py`：可从已生成章节正文中提炼结构化 facts
- `bid_writer/knowledge_assembler.py`：可把知识文档与依赖章节 facts 组装进扩写 prompt

但这两类能力仍停留在“实现机制”层，还没有形成用户可直接理解和操作的“事实卡片”产品模型。当前主要缺口包括：

- 缺少统一的项目级事实卡片库
- 缺少“用户直接录入事实卡片”的入口
- 缺少“提炼后先编辑、再入库”的审阅层
- 缺少扩写前手动勾选卡片并指定“强约束 / 参考”的能力
- 缺少按章节路径保存默认卡片选择方案的能力

用户希望引入“事实卡片”后，把投标方事实的维护与引用从“隐式 prompt 注入”升级为“显式卡片管理 + 显式选择引用”。

## 2. 设计目标

本次设计的目标是：

- 建立统一的项目级事实卡片库
- 支持两种来源：
  - 从现有正文章节手动提炼
  - 用户直接录入
- 提炼结果在正式入库前必须允许用户编辑
- 扩写前由用户手动勾选具体卡片
- 每张被选卡片可设置为“强约束”或“参考”
- 允许用户选择是否把本次勾选结果保存为当前章节的默认方案
- 当启用事实卡片主链路时，章节扩写只使用本次勾选卡片，不再自动注入现有 `knowledge_context`
- 在 GUI 与主配置 YAML 中同时支持该能力

## 3. 非目标

本次设计不包含以下内容：

- 自动在章节保存后无感触发提炼并直接入库
- 复杂语义冲突检测（如不同名称但语义等价的冲突识别）
- 卡片版本历史、回滚、审计流
- 独立的外部卡片库文件
- CLI 卡片管理入口
- 按章节类型或模板批量继承默认卡片方案
- 配置编辑器中的事实卡片可视化编辑页

## 4. 设计结论

### 4.1 选定方案

采用“配置内主存 + 运行时选择态分离”的 `A` 方案。

核心结论如下：

- 事实卡片库是**项目级卡片库**
- 卡片数据保存在主配置文件中
- 扩写时的“强约束 / 参考”属于**本次选择态**，不写回卡片本体
- 默认卡片方案按**具体章节完整路径**保存
- 自动提炼出的卡片是**可编辑快照**，确认保存后进入卡片库
- 同一章节重提炼后确认保存时，替换该章节来源的既有提炼卡片

### 4.2 总体原则

事实卡片链路遵循以下原则：

1. **先审后存**：模型提炼结果必须先进入草稿编辑层，再决定是否入库
2. **显式选择**：扩写只使用用户本次显式勾选的卡片
3. **主数据与使用态分离**：卡片本体不携带章节特定 usage
4. **来源可追溯**：手录与提炼卡片均保留来源元数据
5. **强约束先阻断**：发现冲突的强约束卡片时，先阻止扩写，而不是让模型自行协调

## 5. 数据模型

### 5.1 顶层结构

主配置文件新增顶层区块：

```yaml
fact_cards:
  enabled: true
  cards: []
  chapter_defaults: {}
```

其中：

- `fact_cards.cards`：项目级卡片库
- `fact_cards.chapter_defaults`：按章节路径保存的默认勾选方案

### 5.2 卡片字段

每张事实卡片包含以下字段：

- `id`：稳定唯一标识
- `name`：事实名称
- `content`：事实内容
- `category`：分类，允许用户编辑
- `active`：是否启用
- `source.type`：`manual` / `chapter_extract`
- `source.chapter_path`：若来自章节提炼，则记录来源章节路径
- `source.extraction_instruction`：记录提炼要求摘要
- `created_at`
- `updated_at`

说明：

- 卡片本体不保存“强约束 / 参考”
- 同一张卡片可以在不同章节被不同方式引用

### 5.3 章节默认方案字段

`fact_cards.chapter_defaults` 的 key 为**具体章节完整路径**，value 为默认选中的卡片列表。

每项包含：

- `card_id`
- `usage`：`strong` / `reference`

### 5.4 推荐 YAML 形态

```yaml
fact_cards:
  enabled: true

  cards:
    - id: fact_001
      name: 项目经理
      content: 项目经理由张三担任，具备高级工程师职称，拥有同类项目实施经验。
      category: 人员团队
      active: true
      source:
        type: manual
        chapter_path: ""
        extraction_instruction: ""
      created_at: "2026-04-24T14:30:00+08:00"
      updated_at: "2026-04-24T14:30:00+08:00"

    - id: fact_002
      name: 实施周期
      content: 项目总实施周期为90日历天，分为准备、实施、验收三个阶段。
      category: 实施周期
      active: true
      source:
        type: chapter_extract
        chapter_path: 综合服务项目投标方案 > 项目实施方案 > 项目进度安排
        extraction_instruction: 提炼可供其他章节复用的周期、阶段划分、里程碑信息
      created_at: "2026-04-24T15:10:00+08:00"
      updated_at: "2026-04-24T15:25:00+08:00"

  chapter_defaults:
    综合服务项目投标方案 > 项目实施方案 > 质量保障措施:
      - card_id: fact_001
        usage: reference
      - card_id: fact_002
        usage: strong
```

## 6. 卡片来源与入库流程

### 6.1 来源一：从章节正文手动提炼

提炼入口为用户主动触发，流程如下：

1. 用户在某个已写章节上执行“提炼事实卡片”
2. 用户输入本次提炼要求
3. 系统读取该章节正文，调用模型生成候选卡片草稿
4. 草稿进入编辑态，用户可修改名称、内容、分类，也可删除或补充卡片
5. 用户点击“确认保存”后，正式写入 `fact_cards.cards`

关键约束：

- 若章节当前没有已生成正文，则不能发起提炼
- 若模型返回为空或格式不合法，应尽量转成可编辑草稿；无有效结果时允许用户手动补录
- 未点击“确认保存”前，不能修改正式卡片库

### 6.2 来源二：用户直接录入

第一阶段同时支持两种手录方式：

- 逐条新建：填写“事实名称 + 内容 + 分类”
- 批量录入：一次输入多条，再拆成多张候选卡片

两种方式都先进入草稿层，再由用户确认保存。

### 6.3 草稿层原则

“提炼结果”和“正式卡片库”之间必须存在草稿层。

草稿层承担以下职责：

- 承接模型输出
- 允许用户修正名称、内容、分类
- 支持用户删卡、补卡
- 支持重新提炼覆盖草稿

“重新提炼”只刷新当前草稿，不应直接覆盖已入库卡片。

### 6.4 同章节重提炼的替换规则

用户对同一章节重提炼并点击“确认保存”后，执行“整批替换”。

替换范围限定为：

- `source.type = chapter_extract`
- `source.chapter_path = 当前章节路径`

保存时应尽量遵循以下规则：

- 明显仍为同一张卡片的，优先复用原 `id`
- 新增卡片生成新 `id`
- 被删除的旧卡片，从 `chapter_defaults` 中清理失效引用

这样可以同时满足“整批替换”和“尽量减少默认方案失效”两个目标。

## 7. 扩写前选择与冲突处理

### 7.1 选择原则

章节扩写前新增“事实卡片选择”步骤。

系统行为如下：

- 默认加载当前章节路径对应的 `chapter_defaults`
- 用户可按关键词、分类、来源类型、来源章节筛选卡片
- 用户勾选具体卡片
- 每张勾选卡片可设置为：
  - `strong`
  - `reference`

若本次未勾选任何卡片，则本次扩写不注入事实卡片。

### 7.2 默认方案保存

本次选择完成后，用户可决定是否把当前方案保存为本章节默认值。

具体规则：

- 勾选“记住为本章节默认卡片方案”时，当前选择覆盖写入 `chapter_defaults`
- 不勾选时，本次只临时生效，不改默认方案
- 若用户选择保存且当前无任何卡片，则视为清空该章节默认方案

### 7.3 批量生成边界

当前 GUI 已支持一次选择多个章节后批量生成。

由于事实卡片默认方案按“具体章节路径”保存，而本次手动勾选又是章节级操作，第一阶段对批量生成采用以下收敛策略：

- 单章节生成：允许用户在生成前手动勾选事实卡片，并决定是否保存为该章节默认方案
- 批量生成：不提供一个“整批共用”的临时卡片选择结果
- 批量生成时，每个章节只读取自身已保存的 `chapter_defaults`
- 若某章节没有已保存默认方案，则该章节本次不注入事实卡片

这样可以保持“默认方案绑定到具体章节路径”的一致性，避免在第一阶段引入复杂的多章节卡片选择交互。

### 7.4 强约束冲突处理

第一阶段对冲突采用简单且可解释的硬规则：

- 若同一次扩写中选中了两张 `usage = strong` 的卡片
- 且它们的 `name` 归一化后相同
- 但 `content` 不同

则判定为“强约束冲突”，必须阻止扩写。

阻断时应明确展示：

- 冲突事实名称
- 每张冲突卡片的内容
- 来源信息

第一阶段仅对“强约束 vs 强约束”执行硬阻断，不对“参考 vs 参考”或“强约束 vs 参考”做自动阻断。

## 8. Prompt 注入设计

### 8.1 主链路切换

当 `fact_cards.enabled = true` 且当前章节进入事实卡片扩写链路时：

- 章节扩写只使用本次勾选卡片
- 不再自动注入现有 `knowledge_context`

旧的 `knowledge_*` 配置读取能力与相关代码可阶段性保留，但不再作为事实卡片模式下的默认扩写来源。

### 8.2 Prompt 区块

建议新增专用区块：

```text
## 事实卡片参考
```

内部再拆成两段：

- `### 强约束事实`
- `### 参考事实`

### 8.3 Prompt 语义

对“强约束事实”应明确告知模型：

- 若正文涉及相关信息，必须与这些事实一致
- 不得输出冲突说法
- 不得擅自替换数字、人名、周期、承诺或职责

对“参考事实”应明确告知模型：

- 相关时优先参考吸收
- 若与本章节关联较弱，可不强行展开

同时，应明确限制模型：

- 除本次选中卡片外，不要自行引入未被选中的投标方事实

### 8.4 推荐 Prompt 片段

```text
## 事实卡片参考
以下事实卡片由用户为本次章节扩写显式选定。除以下内容外，不要自行引入未被选中的投标方事实。

### 强约束事实
- [人员团队] 项目经理：项目经理由张三担任，具备高级工程师职称。
- [实施周期] 实施周期：项目总实施周期为90日历天，分三个阶段推进。

要求：若正文涉及上述相关信息，必须保持一致；不得输出冲突表述，不得擅自改写为其他数字、人名、周期或承诺。

### 参考事实
- [服务承诺] 响应机制：提供7×24小时响应支持，重大问题2小时到场。

要求：相关时优先参考吸收；若与本章关联较弱，可不强行写入。
```

## 9. GUI 信息架构

### 9.1 章节提炼入口

复用现有章节右键菜单和操作菜单中的“提炼事实”入口，并升级为“提炼事实卡片”。

弹窗建议包含：

- 来源章节路径
- 提炼要求输入框（默认要求以 placeholder 形式展示，未填写时使用默认要求）
- 当前唯一核心卡片编辑区
- 删除 / 重新提炼
- 确认保存 / 取消

手工新增事实卡片不放在章节提炼工作台内；通过“章节”下拉菜单中的独立“新增事实卡片...”弹窗录入。

### 9.2 生成参数弹窗

复用现有“生成参数设置”弹窗，在“附加扩写要求”下新增“事实卡片”区域。

建议包含：

- 搜索框
- 分类筛选
- 来源筛选
- 卡片列表
- 每张卡片的 `强约束 / 参考` 选择
- “记住为本章节默认卡片方案”复选框

点击“开始生成”前先做冲突校验，冲突则阻止继续。

### 9.3 项目级卡片管理窗口

第一阶段建议新增独立的“事实卡片库管理”窗口，用于：

- 查看全部卡片
- 编辑全部卡片；手动新增单卡走独立“新增事实卡片...”弹窗
- 批量录入
- 编辑 / 停用卡片
- 查看来源信息

这样用户不需要先进入某个章节，也能直接维护项目级卡片库。

### 9.4 配置编辑器边界

第一阶段不要求 `ConfigEditorDialog` 为 `fact_cards` 提供完整的可视化编辑控件。

第一阶段的范围收敛为：

- 事实卡片的主要 GUI 操作入口放在主窗口相关对话框中
- 卡片数据最终仍落到主配置 YAML 中
- 配置编辑器保存现有配置时，应继续透传未受管的 `fact_cards` 区块，不得丢失该数据

后续如果事实卡片配置规模继续增长，再单独为配置编辑器补充专用页面。

## 10. 兼容与迁移边界

### 10.1 与现有知识库链路的关系

当前项目已有：

- `knowledge_files`
- `knowledge_directory`
- `knowledge_context`
- `.bid_writer/chapter_facts.json`

本次设计的边界是：

- 新的事实卡片链路成为扩写主链路
- 旧知识库链路在第一阶段可保留代码兼容，但不再默认进入事实卡片模式下的扩写 prompt
- 旧的章节 facts 缓存可保留为“提炼阶段的技术资产”或迁移期兼容层

### 10.2 配置维护要求

由于本次涉及配置结构变更，后续实现时需要同步维护：

- `docs/config_schema.md`
- `config.example.yaml`
- 相关 `config_*.yaml`
- 对应测试夹具

## 11. 建议实现边界

第一阶段建议把实现范围收敛在以下文件和职责中：

- `bid_writer/config.py`
  - 新增 `fact_cards` 相关配置读取能力
- `bid_writer/gui.py`
  - 增加卡片管理、提炼草稿、扩写前勾选与冲突阻断
- `bid_writer/ai_writer.py`
  - 新增事实卡片 prompt section
  - 在事实卡片模式下停用默认 `knowledge_context` 注入
- `bid_writer/config_editor.py`
  - 保持保存流程对 `fact_cards` 顶层区块的透传，不得因 canonical schema 导出而丢失该数据
- 新增卡片存储与选择态辅助模块
  - 如 `fact_card_store.py`
  - 如 `fact_card_selection.py`

本设计不要求沿用当前 `ChapterFactStore` 的 JSON 结构作为最终卡片库形态。

## 12. 验收标准

至少需要满足以下验收点：

1. 用户可在 GUI 中手动新增、编辑、批量录入事实卡片
2. 用户可从已有章节正文发起“提炼事实卡片”
3. 提炼结果在保存前可编辑，重提炼不会脏写正式卡片库
4. 同章节确认保存后，旧的该章节来源提炼卡片按整批替换处理
5. 用户可在扩写前勾选具体卡片，并设置“强约束 / 参考”
6. 选择互相冲突的强约束卡片时，系统明确阻止扩写
7. 用户可选择是否把本次卡片选择保存为本章节默认方案
8. 扩写 prompt 只注入本次勾选卡片，不再自动带入旧 `knowledge_context`
9. trace 或等价调试信息可回溯本次使用的 `card_id` 与 `usage`

## 13. 设计结论

本次采用“项目级卡片库 + 草稿审阅后入库 + 扩写前显式勾选 + 选择态与主数据分离”的方案。

这样可以同时满足四个核心目标：

- 用户可控：事实先审后用，而不是隐式注入
- 事实一致：强约束卡片冲突时先阻断
- 来源清晰：手录与提炼卡片都可回溯
- 实现收敛：第一阶段即可接入现有 GUI 与配置体系

该方案适合作为后续实现计划的唯一设计基线。
